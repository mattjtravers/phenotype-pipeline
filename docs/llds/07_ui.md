# UI

## Context and Design Philosophy

This component provides a Streamlit web interface for the Business Analyst persona. It is the human-facing entry point to the pipeline: the user uploads SNP data, submits a prediction request, and views the results — phenotype label, confidence score, and top contributing markers.

The UI is a demo interface, not a production web app. It runs locally (`streamlit run`) and optionally deploys to a cloud host. It does not own any ML logic; it dispatches prediction requests to the Lambda API Gateway endpoint.

## User Flow

```
0. On startup: fetch supported phenotype labels from Lambda /labels endpoint
        ↓
1. Upload SNP file (VCF)
        ↓
2. Select target phenotype (dropdown populated from /labels)
        ↓
3. Submit → prediction request dispatched
        ↓
4. Results page:
   - Predicted phenotype label
   - Confidence score (progress bar)
   - Top-20 contributing markers (ranked table + horizontal bar chart of SHAP values)
   - Option to download full PredictionResult as JSON
```

## Phenotype Dropdown

The dropdown options are sourced from the deployed model's `label_encoder.json`. The UI fetches them at startup via a `GET {PHENO_API_ENDPOINT}/labels` call (see `06_deployment.md § Lambda /labels endpoint`). This keeps the UI and the model in lockstep — if a model is redeployed with different phenotype labels, the UI picks them up automatically on next load.

If the `/labels` fetch fails, the dropdown is disabled and an inline error is shown ("Could not load phenotype options — check API endpoint.").

## Execution

The UI always dispatches prediction requests to the Lambda API Gateway endpoint, configured via the `PHENO_API_ENDPOINT` environment variable. There is no local in-process execution mode — inference runs on Lambda only.

## Input Validation

The UI validates the uploaded file before submitting:
- File extension must be `.vcf`
- File size limit: 50 MB (browser-side guard; not a security boundary)
- File must contain exactly one sample — multi-sample VCFs are rejected

Validation errors are shown inline as Streamlit `st.error()` messages.

## Lambda Error Handling

The Lambda returns structured error bodies in the shape `{"error": <code>, "detail": <message>}` (see `docs/llds/06_deployment.md § Error contract`). The UI maps known `error` codes to user-facing messages so that a 503 during a deployment window reads differently from a malformed VCF:

| Code from Lambda | User-facing message (Streamlit `st.error`) |
|---|---|
| `MODEL_UNAVAILABLE` | "The prediction service is starting up or unavailable — please try again in a moment." |
| `INVALID_VCF` | "The uploaded VCF could not be processed (multi-sample file, malformed format, or no usable variants)." |
| `INVALID_INPUT` | "The request was rejected as malformed — please reload and try again." |
| `INFERENCE_FAILED`, `INTERNAL_ERROR` | "The prediction service encountered an error — please try again or contact support." |
| Unknown / missing code, network timeout | Fall through to the generic message in `UI-UI-010`. |

The UI does not retry on its own; the user re-submits manually after seeing the message.

## Decisions & Alternatives

| Decision | Chosen | Alternatives Considered | Rationale |
|---|---|---|---|
| UI framework | Streamlit | Gradio, Flask + React, Jupyter widget | Streamlit requires minimal code and is locally runnable; no JavaScript needed; Gradio is comparable but Streamlit has wider adoption for data apps |
| Prediction integration | HTTP POST to Lambda API Gateway | In-process Python call | Inference runs on Lambda only; no local execution mode |
| Phenotype options source | Lambda `/labels` endpoint (derived from `label_encoder.json`) | Hardcoded UI list, env var at deploy time, S3 read direct from UI | Endpoint keeps UI and model in lockstep without coupling deployments; hardcoded list drifts when a model is retrained on different phenotypes; direct S3 read would require S3 IAM on the UI |
| File input format | VCF only | VCF + CSV | CSV is not used in this project; VCF is the sole data format end-to-end |
